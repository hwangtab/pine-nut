## Potential Conflicts

| 갈등 지점 | 관련 이해관계자 | 완화 전략 |
|----------|---------------|----------|
| **콘텐츠 톤 vs 법적 안전성** | 주민회 ↔ 법률 자문단 | 주민의 감정적 표현을 존중하되, 게시 전 법률 검토 프로세스 수립. "사실 진술"과 "의견 표현"을 구분하는 가이드라인 제공 |
| **개발 속도 vs 콘텐츠 준비** | 개발팀 ↔ 주민회 | 최소 콘텐츠 세트(사진 20장, 인터뷰 3건) 먼저 확보 후 개발 시작. 나머지는 CMS로 점진적 추가 |
| **투명성 vs 보안** | 일반 시민 ↔ 사업자(한수원) | 후원 내역·서명 수는 공개하되, 서명자 개인정보는 철저히 보호. 사업자 측 법적 대응 대비 콘텐츠 백업 체계 |
| **자원봉사 지속성 vs 기능 확장 요구** | 개발팀 ↔ 주민회/환경단체 | Phase별 명확한 범위 설정. Phase 1은 핵심 기능만 집중, 추가 요청은 Phase 2 백로그로 관리 |

## Escalation Path

```
일상적 결정 (콘텐츠 수정, 버그 수정)
  → 개발 리드 / 콘텐츠 담당자가 자율 결정

중요 결정 (새 기능 추가, 디자인 변경, 파트너 연대)
  → 프로젝트 리드 → 주민회 대표 승인

법률/위기 상황 (명예훼손 소송, 개인정보 유출, 사이트 공격)
  → 법률 자문단 즉시 연락 → 주민회 대표 + 프로젝트 리드 공동 대응
  → 필요 시 사이트 일시 비공개 전환 (Vercel 환경변수로 즉시 가능)
```
