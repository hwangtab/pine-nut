## Paper Tigers

| # | 우려 사항 | 실제 위험도가 낮은 이유 | Tiger로 전환되는 조건 |
|---|----------|----------------------|---------------------|
| PT1 | **기술 스택이 너무 복잡하다** — Next.js + Strapi + Supabase 조합이 자원봉사팀에 과할 수 있다 | Next.js는 한국 개발자 풀에서 가장 대중적인 프레임워크, Strapi/Supabase 모두 문서화 잘됨, 대안으로 정적 사이트(Astro + Markdown)로 더 단순화 가능 | 자원봉사 개발자가 1명뿐이고 해당 스택 경험이 없을 때 |
| PT2 | **사이트 해킹/DDoS 공격** — 정치적 반대 세력의 사이버 공격 | Vercel/Cloudflare의 기본 보안이 충분, 정적 사이트 비중이 높아 공격 표면 작음, 서명 DB만 Supabase RLS로 보호 | 대규모 미디어 노출로 정치적 타겟이 될 경우 |
| PT3 | **영문 페이지 품질** — 번역 품질이 낮아 국제 연대에 부정적 | Phase 1에서 영문은 요약 1페이지뿐, 완벽한 번역보다 핵심 전달이 중요, 해외 한인 자원봉사 감수 가능 | 국제 미디어가 집중 보도하기 시작할 때 |

## Elephants in the Room

| # | 불편한 진실 | 왜 논의해야 하는가 | 대화 시작점 |
|---|-----------|------------------|-----------|
| E1 | **발전소 추진 측의 법적 대응 가능성** — 웹사이트 콘텐츠가 명예훼손 또는 업무방해로 법적 분쟁 대상이 될 수 있음 | 이미 일부 주민이 전과자가 된 상황에서, 온라인 콘텐츠는 증거로 활용될 수 있음. 사실 관계 검증과 표현 수위에 대한 법률 가이드라인이 반드시 필요 | "웹사이트에 게시하는 모든 주장에 대해 사실 확인 프로세스와 법률 검토 절차를 정하자. 감정적 표현과 사실적 진술을 구분하는 가이드라인이 필요하다." |
| E2 | **자원봉사 모델의 지속가능성 한계** — 무급 자원봉사에 의존하는 프로젝트는 초기 열정 후 유지보수가 방치되는 패턴이 반복됨 | 4~6주 구축은 가능하지만, 이후 1~2년간 버그 수정, 보안 패치, 콘텐츠 업데이트를 누가 할 것인지 솔직한 대화 필요 | "출시 후 최소 1년간 월 4시간의 유지보수를 책임질 수 있는 사람이 있는가? 없다면 유료 유지보수 예산을 확보하거나, 유지보수가 거의 필요 없는 정적 사이트로 설계해야 한다." |
| E3 | **주민 내부 의견 차이 가능성** — "만장일치 반대"라고 하지만, 웹사이트 콘텐츠·톤·전략에 대해 주민 간 이견이 있을 수 있음 | 웹사이트가 특정 개인이나 소수의 관점만 반영하면 내부 갈등 유발 가능, 콘텐츠 승인 프로세스가 필요 | "주요 콘텐츠(메인 메시지, 인터뷰 게재, 성명서)를 게시하기 전에 주민회 대표 3인 이상의 승인을 받는 절차를 만들자." |

## Go/No-Go Checklist

- [ ] **T1**: 최소 콘텐츠 세트(사진 20장, 인터뷰 3건, 타임라인 이벤트 15개) 확보
- [ ] **T2**: 개인정보 처리방침 법률 검토 완료 (또는 서명 기능 제외 출시 결정)
- [ ] **T3**: 후원금 수령 구조 확정 (또는 계좌이체 안내만으로 출시 결정)
- [ ] **T4**: 핵심 개발자 최소 2명 출시까지 참여 확약
- [ ] **T5**: CMS 운영 담당자 1명 지정 + 매뉴얼 초안 완성
- [ ] **E1**: 콘텐츠 법률 가이드라인 수립 (사실/의견 구분, 표현 수위)
- [ ] **E3**: 콘텐츠 승인 프로세스 주민회 합의
- [ ] Rollback plan: 문제 발생 시 사이트 비공개 전환 절차 확인
- [ ] 도메인 확보 및 SSL 인증서 설정 완료
